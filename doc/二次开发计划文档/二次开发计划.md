# 音乐版权资产管理库（曲库管理）二次开发计划

> **文档版本**: v1.0  
> **编写日期**: 2026-03-19  
> **基线项目**: music-website（Vue 3 + Spring Boot + MySQL 全栈音乐网站）  
> **目标项目**: 音乐版权资产管理库（曲库管理系统）

---

## 1. 项目概述与目标

### 1.1 二次开发背景

基于开源项目 **music-website**（一个面向C端用户的音乐播放网站）进行二次开发，将其改造为面向 **版权代理公司** 的 **音乐版权资产管理库**。原项目提供了音乐播放、用户社交（评论/收藏/评分）、后台内容管理等基础能力，目标项目需在此基础上扩展为企业级资产管理系统。

### 1.2 核心改造目标

| 维度 | 原项目能力 | 目标项目要求 | 改造类型 |
|------|-----------|-------------|---------|
| 用户体系 | 单一管理员 + 普通用户 | 多角色RBAC + 组织结构 | **重构** |
| 内容模型 | 歌手→歌曲→歌单 | 项目→物料（音频/歌词/谱子/合同） | **重构** |
| 权限控制 | 简单登录验证 | 系统/组织/项目/物料/功能五级权限 | **新增** |
| 搜索系统 | 简单模糊搜索 | 多维度标签 + 高级搜索 + 排序 | **增强** |
| 文件管理 | 单文件上传(20MB限制) | 批量/拖拽/断点续传/文件夹上传 | **增强** |
| 数据看板 | 简单统计图表 | 自定义看板 + 多图表类型 | **增强** |
| 网站管理 | 无 | 主题/图标/字体定制化 | **新增** |
| 小程序 | 无 | 微信小程序（预留接口） | **新增** |
| 状态管理 | 无 | 已发行/未发行/暂缓发行 | **新增** |

## 3. 数据库改造方案

### 3.1 保留并修改的表

#### consumer → user（用户表重构）

```diff
- consumer (id, username, password, sex, phone_num, email, birth, introduction, location, avator, create_time, update_time)
+ user (id, username, password, nickname, sex, phone_num, email, birth, introduction, location, avatar,
+       org_id, role_id, status, skill_tags, role_tags, create_time, update_time, last_login_time)
```

新增字段：`org_id`(所属组织)、`role_id`(角色)、`status`(启用/禁用)、`skill_tags`(技能标签)、`role_tags`(角色标签)、`last_login_time`

#### admin → 合并到user表

管理员不再单独建表，通过角色区分。

#### song → 保留核心字段，扩展

```diff
- song (id, singer_id, name, introduction, create_time, update_time, pic, lyric, url)
+ song (id, name, artist_ids, introduction, create_time, update_time, release_time, pic, lyric, url,
+       project_id, status, genre, language, duration, bpm, key_signature, isrc)
```

新增字段：`project_id`(关联项目)、`status`(已发行/未发行/暂缓)、`genre`(风格)、`language`(语言)、`duration`(时长)、`bpm`、`key_signature`(调式)、`isrc`(国际标准录音编码)

### 3.2 需要新增的表

| 表名 | 用途 | 核心字段 |
|------|------|---------|
| **organization** | 组织结构 | id, name, parent_id, level, path, sort, status, create_time |
| **role** | 角色表 | id, name, code, description, type(系统/自定义), status |
| **permission** | 权限表 | id, name, code, type(菜单/按钮/数据), parent_id, path, sort |
| **role_permission** | 角色-权限关联 | id, role_id, permission_id |
| **user_role** | 用户-角色关联 | id, user_id, role_id, scope_type, scope_id |
| **permission_template** | 权限模板 | id, name, description, permissions_json |
| **project** | 歌曲项目 | id, name, artist_ids, description, status, cover_pic, tags, created_by, org_id, create_time, update_time |
| **material** | 物料表 | id, project_id, name, type(音频/歌词/谱子/合同/图片), file_url, file_size, file_format, description, tags, uploaded_by, status, create_time |
| **tag** | 标签表 | id, name, category, parent_id, sort, create_time |
| **project_tag** | 项目-标签关联 | id, project_id, tag_id |
| **material_tag** | 物料-标签关联 | id, material_id, tag_id |
| **artist** | 艺术家表 | id, name, type(词作者/曲作者/歌手/制作人), sex, pic, introduction, contact_info |
| **project_artist** | 项目-艺术家关联 | id, project_id, artist_id, role(词/曲/演唱/编曲) |
| **operation_log** | 操作日志 | id, user_id, action, target_type, target_id, detail, ip, create_time |
| **site_config** | 网站配置 | id, config_key, config_value, config_type, description |
| **dashboard** | 看板配置 | id, name, layout_json, owner_id, is_public, create_time |
| **user_tag** | 用户标签 | id, user_id, tag_name, tag_type(技能/角色) |

### 3.3 数据库ER关系概览

```
user ──1:N──► project (创建者)
user ──N:1──► organization (所属组织)
user ──N:N──► role (通过 user_role)
role ──N:N──► permission (通过 role_permission)

project ──1:N──► material (物料)
project ──N:N──► tag (通过 project_tag)
project ──N:N──► artist (通过 project_artist)

material ──N:N──► tag (通过 material_tag)

organization ──自引用──► organization (parent_id 树形结构)
```

### 3.4 数据迁移策略

1. **保留原表结构**作为备份（添加 `_bak` 后缀）
2. **创建新表**按新设计建表
3. **数据迁移脚本**：将 `consumer` → `user`，`singer` → `artist`，`song` → `song`+`project`
4. **验证迁移**：编写校验SQL确认数据完整性

## 4. 后端API改造方案

### 4.1 鉴权系统改造

**原系统**：无Token机制，仅靠session判断登录状态  
**目标系统**：JWT + RBAC

| 改造内容 | 说明 |
|---------|------|
| 引入Spring Security | 集成JWT认证过滤器 |
| Token管理 | Access Token(2h) + Refresh Token(7d) |
| 权限注解 | 自定义`@RequirePermission`注解 |
| 数据权限 | 基于组织和项目的数据隔离 |

新增API：

```
POST   /auth/login              # 统一登录（替代原admin/login和user/login）
POST   /auth/logout             # 登出
POST   /auth/refresh            # 刷新Token
GET    /auth/info               # 获取当前用户信息和权限
```

### 4.2 新增API模块

#### 组织管理 `/api/org`
```
GET    /api/org/tree            # 获取组织结构树
POST   /api/org                 # 创建组织

GET    /api/org/{id}/users      # 获取组织下用户
```

## 8. 开发测试流程

### 8.1 开发流程规范

```
需求确认 → 技术方案评审 → 编码 → Code Review → 单元测试 → 集成测试 → 验收测试 → 上线
```

#### 8.1.1 分支管理策略（Git Flow）

```
main          ─────────────────────────────────► (稳定版本)
  └── develop ─────────────────────────────────► (开发主线)
        ├── feature/auth-jwt       → 鉴权功能分支
        ├── feature/project-mgmt   → 项目管理分支
        ├── feature/material-mgmt  → 物料管理分支
        ├── feature/search-system  → 搜索系统分支
        └── feature/dashboard      → 看板系统分支
```

#### 8.1.2 代码提交规范

```
feat: 新功能
fix: 修复bug
docs: 文档更新
style: 代码格式（不影响功能）
refactor: 重构
test: 测试相关
chore: 构建/工具链相关
```

#### 8.1.3 Code Review规范

- 每个PR至少1人Review
- 核心模块（鉴权/权限）至少2人Review
- Review关注点：安全性、性能、代码规范、边界条件

## 10. 风险与应对措施

| 风险 | 等级 | 影响 | 应对措施 |
|------|------|------|---------|
| UI库迁移(EP→ADV)工作量大 | 高 | 延期2-3周 | 分阶段迁移，第一阶段保持EP |

| 微信小程序开发 | 低 | 范围蔓延 | 第一版仅预留API，不实际开发小程序 |
