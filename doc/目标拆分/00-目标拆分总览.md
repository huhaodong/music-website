# 音乐版权资产管理库 -- 目标拆分总览

> **文档版本**: v1.0
> **编写日期**: 2026-03-19
> **开发模式**: 测试驱动开发（TDD）
> **基线项目**: music-website（Vue 3 + Spring Boot + MySQL）

---

## 1. 拆分原则

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **项目可用性优先** | 每个里程碑结束后，项目必须可以正常编译、启动、运行，不破坏现有可用功能 |
| **测试驱动开发** | 先写测试用例（红），再写实现代码（绿），最后重构（蓝）。每个里程碑必须附带完整的测试验收标准 |
| **增量交付** | 每个里程碑是一个可独立验收的交付单元，后续里程碑在前序基础上增量演进 |
| **最小可用改动** | 优先改造必须变更的部分，尽量复用原有代码，避免不必要的重构 |
| **向后兼容** | 新增功能不破坏已有功能，数据库变更通过增量迁移脚本而非破坏性修改 |

### 1.2 TDD 工作流

```
每个功能点的开发流程：

1. 编写失败的测试用例（RED）
   ├── 后端：JUnit 5 + Mockito 单元测试
   ├── 后端：Spring MockMvc 接口测试
   ├── 前端：Vitest 工具函数测试
   └── 前端：Vue Test Utils 组件测试

2. 编写最小实现代码使测试通过（GREEN）
   └── 仅编写刚好让测试通过的代码

3. 重构优化（REFACTOR）
   ├── 消除重复代码
   ├── 优化命名和结构
   └── 确保所有测试仍然通过

4. 里程碑验收
   ├── 所有单元测试通过
   ├── 所有集成测试通过
   ├── 项目可正常启动运行
   └── 验收清单逐项确认
```

## 2. 里程碑总览

共拆分为 **9 个里程碑（M0 - M8）**，预计总工期 **16 周**。

```
M0 ──► M1 ──► M2 ──► M3 ──► M4 ──► M5 ──► M6 ──► M7 ──► M8
基线    鉴权    权限    业务上  业务下  前端    搜索    增强    上线
(1周)  (2周)  (2周)  (2周)  (2周)  (2.5周) (2周)  (1.5周) (1周)
```

### 2.1 里程碑一览表

| 里程碑 | 名称 | 周期 | 核心目标 | 验收门槛 |
|--------|------|------|----------|----------|
| **M0** | 项目基线确认与TDD基础设施搭建 | 第1周 | 确认原项目可运行，搭建测试框架，建立CI流程 | 原项目通过冒烟测试；测试框架可运行 |
| **M1** | 数据库演进与JWT鉴权体系 | 第2-3周 | 增量建表，JWT认证替换Session，统一登录接口 | 新旧用户均可JWT登录；原有功能不受影响 |
| **M2** | RBAC权限系统与组织用户管理 | 第4-5周 | 角色/权限/组织CRUD，用户表重构，数据隔离 | 不同角色用户权限隔离生效 |
| **M3** | 艺术家管理与项目管理 | 第6-7周 | 艺术家CRUD，项目CRUD，状态机管理 | 项目全生命周期可管理 |
| **M4** | 物料管理与标签系统 | 第8-9周 | 物料上传/预览/CRUD，标签体系，关联管理 | 物料可上传/预览/管理/打标签 |
| **M5** | 前端核心页面改造 | 第10-11.5周 | 前台首页/搜索页/播放器改造，后台管理页面开发 | 前后台页面可正常使用 |
| **M6** | 搜索系统与数据看板 | 第12-13周 | 高级搜索，数据统计看板 | 多条件搜索可用；看板展示正确 |
| **M7** | 增强功能与网站定制化 | 第14-15周前半 | 分片上传、操作日志、网站主题定制 | 大文件可分片上传；主题可切换 |
| **M8** | 集成测试、性能优化与部署上线 | 第15周后半-16周 | 全量测试、性能调优、生产部署 | 所有验收指标达标；生产环境可访问 |

### 2.2 依赖关系图

```
M0（基线）
 └──► M1（鉴权）
       └──► M2（权限）
             ├──► M3（艺术家+项目）
             │     └──► M4（物料+标签）
             │           └──► M5（前端页面）
             │                 ├──► M6（搜索+看板）
             │                 └──► M7（增强功能）
             │                       └──► M8（测试+部署）
             └──────────────────────────────────┘
```

### 2.3 风险缓冲

- 每个里程碑预留 **10%-15%** 缓冲时间
- M1/M2 为高风险里程碑（基础设施变更），预留较多缓冲
- 如果进度提前，可将 M6/M7 的功能提前实现

## 3. 测试策略总览

### 3.1 各里程碑测试层级

| 里程碑 | 单元测试 | 接口测试 | 集成测试 | E2E测试 |
|--------|----------|----------|----------|---------|
| M0 | 搭建框架 | 搭建框架 | - | - |
| M1 | 鉴权Service | Auth API | 登录全流程 | - |
| M2 | 权限Service | Role/Org API | 权限隔离链路 | - |
| M3 | 项目Service | Project/Artist API | 项目CRUD链路 | - |
| M4 | 物料Service | Material/Tag API | 上传→管理链路 | - |
| M5 | 前端工具函数 | - | - | 关键页面流程 |
| M6 | 搜索/统计Service | Search/Dashboard API | 搜索全链路 | - |
| M7 | 配置Service | System API | 文件上传链路 | - |
| M8 | 补充遗漏 | 全量回归 | 全量回归 | 全量E2E |

### 3.2 测试工具链

| 层级 | 后端工具 | 前端工具 |
|------|---------|---------|
| 单元测试 | JUnit 5 + Mockito + H2(内存DB) | Vitest + Vue Test Utils |
| 接口测试 | Spring MockMvc + REST Assured | - |
| 集成测试 | Testcontainers(MySQL+Redis+MinIO) | - |
| E2E测试 | - | Playwright / Cypress |
| 性能测试 | JMeter / k6 | Lighthouse |
| 安全测试 | OWASP ZAP / sqlmap | - |

### 3.3 持续集成门禁

每次代码提交必须通过以下门禁：

```
提交代码 → 编译检查 → 单元测试 → 代码覆盖率检查(≥60%) → 通过 → 合并
                                                      → 未通过 → 阻断合并
```
